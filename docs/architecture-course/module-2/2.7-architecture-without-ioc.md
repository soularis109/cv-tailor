# 2.7 — Архітектура без IoC-контейнера

Не завжди є можливість налаштувати повноцінний DI через Context.
Ця тема показує як **зберегти чистоту архітектури** навіть коли використовуєш прості хуки або React Query.

---

## Публічне API модуля: barrel файли

**Barrel файл** (`index.ts`) — єдина точка входу в модуль.
Зовнішній код знає тільки що дозволяє barrel, а не внутрішню структуру.

### Проблема без barrel

```tsx
// Зовнішній код знає про внутрішню структуру
import { Pipeline } from "../features/pipeline/Pipeline";
import { usePipelineData } from "../features/pipeline/usePipelineData";
import { PipelineStats } from "../features/pipeline/PipelineStats";
import { statusClass } from "../features/pipeline/utils";
// 10 рядків імпортів з деталей реалізації
```

**Проблема**: ти перейменував `usePipelineData.ts` → `usePipeline.ts`? Треба оновити всіх споживачів.

### Рішення: barrel index.ts

```ts
// client/src/features/pipeline/index.ts
// Тільки те що призначено для зовнішнього використання

export { Pipeline } from "./Pipeline";
export { usePipelineData } from "./usePipelineData";
// Не експортуємо: statusClass, PipelineRow, internal utils
```

```tsx
// Зовнішній код — тільки з барелю
import { Pipeline, usePipelineData } from "../features/pipeline";
// 1 рядок замість 4+
```

Тепер можеш перейменовувати внутрішні файли — barrel файл оновлюється в одному місці.

## Dependency Scopes

Не всі залежності живуть однаково довго:

| Scope | Lifetime | Приклад |
|-------|---------|---------|
| **Singleton** | Весь час роботи | ApiClient, EventEmitter |
| **Module** | Поки компонент змонтований | Store конкретної фічі |
| **Request** | Один HTTP-запит | RequestId, AbortController |
| **Transient** | Кожен виклик | Нова дата, UUID |

```tsx
// Singleton scope — живе весь час програми
// main.tsx
const apiClient = new ApiClient(BASE_URL);  // один екземпляр

ReactDOM.createRoot(root).render(
  <ApiClientContext.Provider value={apiClient}>
    <App />
  </ApiClientContext.Provider>
);

// Module scope — живе поки компонент існує
function PipelineFeature() {
  // Новий store на кожен mount цього компонента
  const [store] = useState(() => createLocalPipelineStore());

  return (
    <PipelineStoreContext.Provider value={store}>
      <Pipeline />
    </PipelineStoreContext.Provider>
  );
}

// Request scope — AbortController
function useApplicationData(id: string) {
  useEffect(() => {
    const controller = new AbortController();  // transient: новий кожен раз

    api.getApplicationData(id, { signal: controller.signal })
      .then(setData)
      .catch(e => {
        if (e.name !== "AbortError") setError(e.message);
      });

    return () => controller.abort();  // cleanup при unmount
  }, [id]);
}
```

## DI через функції-фабрики (без класів)

Класи в JavaScript нести overhead і ускладнюють composability.
Функції-фабрики — легший альтернатив:

### Клас vs Функція-фабрика

```ts
// ❌ Клас — необхідний this, instanceof, extends
class ApplicationService {
  constructor(private readonly api: ApiClient) {}

  async getApplicationData(id: string): Promise<ApplicationData> {
    return this.api.get(`/applications/${id}/data`);
  }
}
const service = new ApplicationService(apiClient);

// ✅ Функція-фабрика — проста композиція
function createApplicationService(api: ApiClient): ApplicationService {
  return {
    async getApplicationData(id) {
      return api.get(`/applications/${id}/data`);
    },
    async patchApplication(id, patch) {
      return api.patch(`/applications/${id}`, patch);
    },
  };
}
const service = createApplicationService(apiClient);
```

**Переваги функцій-фабрик**:
- Легко мокати (просто об'єкт)
- Немає проблем з `this`
- TypeScript виводить типи автоматично
- Composable: `createLoggingService(createApplicationService(api))`

### Composable services (декоратор без класів)

```ts
// Logging decorator — без класів
function withLogging<T extends ApplicationService>(
  service: T,
  logger: Logger
): T {
  return {
    ...service,
    async getApplicationData(id) {
      logger.info(`[ApplicationService] getApplicationData(${id})`);
      const start = Date.now();
      const result = await service.getApplicationData(id);
      logger.info(`[ApplicationService] done in ${Date.now() - start}ms`);
      return result;
    },
  };
}

// Composition Root
const service = withLogging(
  createApplicationService(apiClient),
  console
);
```

## Рефакторинг на React Query: що змінюється

React Query — це de-facto стандарт для Server State.
Але він не замінює Service Layer — він **змінює де living state**, не де живе логіка.

### Поточний паттерн cv-tailor (useState + useEffect)

```tsx
// Завантаження Master CV — App.tsx рядки 142–161
useEffect(() => {
  setMasterLoading(true);
  setMaster(null);
  api.getMasterCv(activeProfile)
    .then(cv => { setMaster(cv); })
    .catch(e => { setMasterError(e.message); })
    .finally(() => setMasterLoading(false));
}, [activeProfile]);
```

### Той самий код з React Query

```tsx
import { useQuery } from "@tanstack/react-query";

function useMasterCv(profile: string) {
  return useQuery({
    queryKey: ["master-cv", profile],  // кеш-ключ включає profile
    queryFn: () => api.getMasterCv(profile),
    staleTime: 60_000,  // свіжий 1 хвилину
    retry: 2,           // 2 повторних спроби при помилці
  });
}

// В компоненті:
const { data: master, isLoading: masterLoading, error } = useMasterCv(activeProfile);
// Зникло: 4 useState, 1 useEffect, 10 рядків бойлерплейту
```

### DI + React Query: зберігаємо абстракцію

```tsx
// ✅ React Query через сервіс (не через api.ts напряму)
function useMasterCv(profile: string) {
  const service = useMasterCvService();  // DI через Context

  return useQuery({
    queryKey: ["master-cv", profile],
    queryFn: () => service.getMasterCv(profile),  // ← сервіс, не api
  });
}
```

Переваги: при зміні реалізації (api → GraphQL) — змінюємо тільки `ApiMasterCvService`, не хук.

## Самописний DI-контейнер (Poor Man's DI)

Коли Context є «занадто» або потрібна більша гнучкість — простий контейнер:

```ts
// client/src/services/container.ts

type Factory<T> = () => T;
type LifetimeMap = Map<string, unknown>;

class Container {
  private singletons: LifetimeMap = new Map();
  private factories: Map<string, Factory<unknown>> = new Map();

  register<T>(key: string, factory: Factory<T>, singleton = true): void {
    this.factories.set(key, factory);
    if (!singleton) return;
    this.singletons.delete(key);  // reset якщо перереєструємо
  }

  resolve<T>(key: string): T {
    if (this.singletons.has(key)) {
      return this.singletons.get(key) as T;
    }
    const factory = this.factories.get(key);
    if (!factory) throw new Error(`Service not registered: ${key}`);
    const instance = factory() as T;
    this.singletons.set(key, instance);
    return instance;
  }
}

// Composition Root
export const container = new Container();

container.register("ApiClient", () => new ApiClient(import.meta.env.VITE_API_URL));
container.register("ApplicationService", () =>
  createApplicationService(container.resolve("ApiClient"))
);

// Використання (в хуках, де Context недоступний)
const service = container.resolve<ApplicationService>("ApplicationService");
```

**Коли Poor Man's DI ок**: утиліти, middleware, коли Context overhead надмірний.
**Коли не ок**: у компонентах — тоді не видно залежностей, важко тестувати.
