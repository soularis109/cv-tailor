# 2.6 — Production-фічі: доводимо до бойового стану

Учбовий проєкт → production розрізняє наявність:
- Optimistic UI (швидкий відклик)
- Правильна обробка помилок
- Ролева модель
- Cross-domain комунікація
- Архітектурні межі що не дають їх порушувати

---

## Optimistic UI

**Ідея**: оновити UI **одразу** (не чекаючи API), і відкотити якщо запит провалиться.

### Поточна реалізація в cv-tailor (Pipeline delete)

```tsx
// client/src/App.tsx — вже є примітивний optimistic delete

const [pendingDeletes] = useState<Map<string, PendingDelete>>(() => new Map());

function handleDelete(id: string) {
  // 1. Оновлюємо UI одразу (ще не видалено на сервері)
  setApps((prev) => prev.filter((a) => a.id !== id));

  // 2. Показуємо "Undo" toast на 5 секунд
  const timeoutId = setTimeout(async () => {
    pendingDeletes.delete(id);
    await api.deleteApplication(id);  // 3. Тільки тут реальний DELETE
  }, 5000);

  pendingDeletes.set(id, { app: deletedApp, timeoutId });

  showToast(`Removed ${app.company}`, "info", {
    action: { label: "Undo", onClick: () => handleUndoDelete(id) },
    duration: 5000,
  });
}
```

**Але є проблема**: якщо `api.deleteApplication` кидає помилку — UI вже оновлений, rollback не відбувається.

### Правильний optimistic з rollback

```tsx
async function deleteApp(id: string) {
  // Зберігаємо стан ДО змін (для rollback)
  const previousApps = apps;

  // Optimistic update
  setApps(prev => prev.filter(a => a.id !== id));

  try {
    await api.deleteApplication(id);
  } catch (error) {
    // Rollback при помилці
    setApps(previousApps);
    showToast("Could not delete application. Try again.", "error");
  }
}
```

### Optimistic update з React Query

```tsx
const deleteApp = useMutation({
  mutationFn: (id: string) => api.deleteApplication(id),

  // Optimistic update
  onMutate: async (id) => {
    // Скасуй поточні queries щоб уникнути race condition
    await queryClient.cancelQueries({ queryKey: ["applications"] });

    // Зберігаємо попередні дані для rollback
    const previousApps = queryClient.getQueryData(["applications"]);

    // Optimistically update
    queryClient.setQueryData(["applications"], (old: Application[]) =>
      old.filter(a => a.id !== id)
    );

    return { previousApps };  // поверни для onError
  },

  onError: (_err, _id, context) => {
    // Rollback при помилці
    queryClient.setQueryData(["applications"], context!.previousApps);
  },

  onSettled: () => {
    // Завжди рефетч для синхронізації
    queryClient.invalidateQueries({ queryKey: ["applications"] });
  },
});
```

## Обробка помилок: Fail Fast

**Fail Fast** — система падає швидко і гучно при невалідних даних, замість того щоб тихо продовжувати з некоректним станом.

### Антипатерн: тихе ковтання помилок

```tsx
// ❌ Тихе ковтання — в cv-tailor є такі місця
api.runAtsCheck(application.id).catch(() => {
  // silent — user can retry
  setAtsLoading(false);
});

// Проблема: ти не знаєш чи провалився ATS чи просто timeout
// Моніторинг не бачить помилку
```

### Fail Fast для різних контекстів

```tsx
// 1. Для критичних операцій — показуй помилку і логуй
async function runAtsCheck(id: string) {
  setAtsLoading(true);
  try {
    const result = await service.runAtsCheck(id);
    setAtsResult(result);
  } catch (error) {
    const message = error instanceof Error ? error.message : "ATS check failed";
    showToast(message, "error");
    console.error("[ATS Check]", error);  // для моніторингу
  } finally {
    setAtsLoading(false);
  }
}

// 2. Error Boundary — для несподіваних помилок рендерингу
class ErrorBoundary extends React.Component<
  { children: ReactNode; fallback: ReactNode },
  { hasError: boolean }
> {
  state = { hasError: false };

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  componentDidCatch(error: Error, info: ErrorInfo) {
    console.error("Render error:", error, info);
    // trackError(error);  // Sentry, Datadog etc.
  }

  render() {
    return this.state.hasError ? this.props.fallback : this.props.children;
  }
}

// 3. Validation guard — Fail Fast на вході
function useApplicationDetail(id: string | null) {
  if (!id) {
    // Fail Fast: не починай завантаження якщо id немає
    return { data: null, loading: false, error: "No application selected" };
  }
  // ...
}
```

## Ролева модель: компонент Can

**Задача**: різні користувачі мають різні права. Але зараз в cv-tailor немає ролей — це важлива production-фіча.

### Концепція: компонент Can

```tsx
// Декларативна перевірка прав прямо в JSX
<Can do="delete" on="application">
  <button onClick={() => deleteApp(id)}>Delete</button>
</Can>

// Якщо немає прав — кнопка просто не рендериться
```

### Реалізація через DIP

```ts
// Абстракція — правила прав
export interface PermissionService {
  can(action: string, resource: string): boolean;
}

// Реалізація — поки що одна роль (owner)
export const ownerPermissions: PermissionService = {
  can(action, resource) {
    // Власник може все
    return true;
  },
};

// Майбутня реалізація — читачу
export const viewerPermissions: PermissionService = {
  can(action, resource) {
    if (action === "delete") return false;
    if (action === "edit") return false;
    return true;  // читати можна все
  },
};
```

```tsx
// Контекст
const PermissionContext = createContext<PermissionService>(ownerPermissions);
export function usePermissions() { return useContext(PermissionContext); }

// Компонент Can
interface CanProps {
  do: string;
  on: string;
  children: ReactNode;
  fallback?: ReactNode;
}

export function Can({ do: action, on: resource, children, fallback = null }: CanProps) {
  const permissions = usePermissions();
  return permissions.can(action, resource) ? <>{children}</> : <>{fallback}</>;
}

// Використання в Pipeline
<Can do="delete" on="application" fallback={<span className="text-muted">—</span>}>
  <button onClick={() => onDelete(app.id)}>Delete</button>
</Can>
```

## Facade: cross-domain комунікація

**Facade** — простий інтерфейс над складною підсистемою.
Особливо корисний коли одна операція торкається **декількох Bounded Context** одночас.

### Проблема: TailorWorkflow торкається трьох доменів

```tsx
// App.tsx — зараз: оркестрація розмазана по компоненту
async function runTailor() {
  // Domain: AI Services
  const analysis = await api.analyzeJob(jobText);
  const result = await api.tailor(jobText, master);

  // Domain: Job Hunt (Pipeline)
  setApps(prev => [...prev, result.application]);

  // Domain: AI Services (Company Brief)
  api.getCompanyBrief(result.application.id);

  // Domain: AI Services (ATS)
  api.runAtsCheck(result.application.id);
}
```

### Рішення: TailoringFacade

```ts
// client/src/services/TailoringFacade.ts
// Facade координує між AI Services і Job Hunt Context

export interface TailoringFacade {
  runFullTailor(
    jobText: string,
    cv: MasterCv,
    options: TailoringOptions
  ): Promise<TailorResponse>;
}

export function createTailoringFacade(
  tailoringService: TailoringService,
  applicationService: ApplicationService,
  events: AppEventEmitter,
): TailoringFacade {
  return {
    async runFullTailor(jobText, cv, options) {
      // 1. AI Services: аналіз і tailoring
      const result = await tailoringService.tailorCv(jobText, cv, options);

      // 2. Job Hunt: зберегти заявку
      // (через події — Facade не знає як Pipeline зберігає)
      events.emit("tailor:completed", { result });

      // 3. AI Services: фонові задачі (fire and forget)
      tailoringService.generateCompanyBrief(result.application.id).catch(console.error);
      tailoringService.runAtsCheck(result.application.id).catch(console.error);

      return result;
    },
  };
}
```

**Тепер `useTailorWorkflow` викликає тільки Facade** — не знає про деталі координації.

## ESLint для архітектурних меж

Щоб межі між шарами не порушувались випадково — можна автоматизувати перевірку.

```js
// .eslintrc.cjs
module.exports = {
  rules: {
    // "no-restricted-imports" забороняє певні import-и в певних директоріях
    "no-restricted-imports": ["error", {
      patterns: [
        {
          // Компоненти не можуть імпортувати api.ts прямо
          group: ["../api", "../../api", "../../../api"],
          message: "Components must use services via Context, not api.ts directly",
        },
        {
          // Shared не може імпортувати features
          group: ["*/features/*"],
          message: "Shared layer cannot depend on features layer",
        },
      ],
    }],
  },
  overrides: [
    {
      // Виняток для services/ — вони можуть
      files: ["src/services/**"],
      rules: { "no-restricted-imports": "off" },
    },
  ],
};
```

## ДЗ 13 — Optimistic delete з rollback

### Завдання

Поточний optimistic delete в `App.tsx` не має rollback при помилці API.
Твоє завдання:
1. Знайди функцію `handleDelete` в `App.tsx`
2. Збережи стан apps **до** видалення (для rollback)
3. Додай `try/catch` навколо `api.deleteApplication`
4. В `catch` — відновлюй apps і показуй toast з помилкою
5. В хорошому сценарії — залишаємо існуючу Undo-логіку
