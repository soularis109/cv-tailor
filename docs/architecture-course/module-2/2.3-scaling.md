# 2.3 — Масштабування архітектури при зростанні складності

Шарова архітектура з Модулю 2.1 вирішує 80% проблем.
Але при зростанні фіч з'являються нові: **границі між доменами**, **комунікація між модулями**, **ізоляція змін**.

## Еволюція структури папок

### Фаза 1: Невеликий проєкт (~5 компонентів)
```
src/
  components/
  hooks/
  utils/
  api.ts
```
Ок. Все поміщається в голові.

### Фаза 2: Середній проєкт (~15 компонентів) — cv-tailor зараз
```
src/
  components/    ← 14 файлів, різні домени перемішані
  hooks/         ← 1 файл
  utils/         ← 3 файли
  api.ts         ← один файл для всіх ендпоінтів
```
Починаються проблеми: де шукати «логіку tailoring»? В `components/App.tsx`? В `api.ts`?

### Фаза 3: Великий проєкт (~30+ компонентів) — ціль cv-tailor
```
src/
  features/
    tailor/          ← фіча tailoring
      TailorForm.tsx
      useTailorWorkflow.ts
      TailoringService.ts
      index.ts         ← публічне API фічі
    pipeline/        ← фіча pipeline
      Pipeline.tsx
      usePipelineData.ts
      PipelineService.ts
      index.ts
    application-detail/
      ApplicationDetailPanel.tsx
      useApplicationDetail.ts
      index.ts
  shared/            ← спільне між фічами
    components/ui/
    services/
    types/
    utils/
  main.tsx
```

## FSD — Feature Sliced Design (адаптований)

FSD — стандарт структури великих React-проєктів. Ключова ідея: **вертикальні слайси (фічі)** замість **горизонтальних шарів**.

```
layers (зверху вниз — напрямок залежностей):
  app/        ← Composition Root, роутинг, провайдери
  pages/      ← Сторінки (комбінують widgets + features)
  widgets/    ← Великі self-contained секції UI
  features/   ← Фічі з бізнес-логікою
  entities/   ← Бізнес-сутності (Application, MasterCv)
  shared/     ← Утиліти, UI-кіт, API-клієнт
```

**Правило FSD**: шар може залежати тільки від шарів **нижче** нього.
`features/` може використовувати `entities/` і `shared/`, але не `pages/` або `widgets/`.

### Де cv-tailor знаходиться в FSD

```
cv-tailor FSD mapping:
  app/      ← main.tsx, App.tsx (providers)
  widgets/  ← ApplicationDetailPanel (велика self-contained секція)
  features/ ← TailorWorkflow, Pipeline (фічі з логікою)
  entities/ ← Application, MasterCv, TailoredCv (бізнес-сутності)
  shared/   ← api.ts, utils/, components/ui/
```

## Bounded Context: де домени починаються і закінчуються

**Bounded Context** — термін з DDD (Domain-Driven Design).
Це зона коду де певна **модель має сенс** і за межами якої вона стає чужою.

### Три Bounded Context в cv-tailor

```
┌─────────────────────────────┐
│   CV Management Context     │
│                             │
│   MasterCv, CvProfile       │
│   MasterCvDrawer            │
│   MasterCvService           │
│                             │
│   «Я управляю резюме»       │
└─────────────────────────────┘

┌─────────────────────────────┐
│   Job Hunt Context          │
│                             │
│   Application, Status       │
│   Pipeline, PipelineService │
│   ApplicationDetailPanel    │
│                             │
│   «Я відстежую заявки»      │
└─────────────────────────────┘

┌─────────────────────────────┐
│   AI Services Context       │
│                             │
│   JobAnalysis, TailoredCv   │
│   TailorWorkflow            │
│   TailoringService          │
│   AtsCheckService           │
│                             │
│   «Я оброблю вакансію AI»   │
└─────────────────────────────┘
```

**Важливо**: одна і та сама концепція може мати різні моделі в різних контекстах.

Наприклад, `Application` в **Job Hunt Context** — це `{ id, company, status, fitScore }`.
У **AI Services Context** — це `{ jobText, analysis, tailored }`.
Це **не одна модель** — це дві різні проекції однієї бізнес-концепції. І правильно тримати їх окремо.

## Комунікація між модулями через Event Emitter

Коли два Bounded Context мають взаємодіяти — робиться це через **події** (events), не через прямі виклики.

**Проблема прямого виклику**:
```tsx
// ❌ TailorWorkflow знає про Pipeline — це coupling між двома контекстами
function useTailorWorkflow() {
  const pipeline = usePipelineStore();  // ← залежність на чужий контекст!

  async function runTailor() {
    const result = await tailoringService.tailor(...);
    pipeline.addApp(result.application);  // ← TailorWorkflow змінює Pipeline
  }
}
```

**Рішення через EventEmitter**:
```ts
// client/src/shared/events.ts — спільна шина подій
type EventMap = {
  "application:created": { application: Application };
  "tailor:completed": { result: TailorResponse };
  "application:deleted": { id: string };
};

type EventHandler<T> = (payload: T) => void;
const listeners = new Map<string, Set<EventHandler<unknown>>>();

export const appEvents = {
  emit<K extends keyof EventMap>(event: K, payload: EventMap[K]) {
    listeners.get(event)?.forEach(fn => fn(payload as unknown));
  },
  on<K extends keyof EventMap>(event: K, handler: EventHandler<EventMap[K]>) {
    if (!listeners.has(event)) listeners.set(event, new Set());
    listeners.get(event)!.add(handler as EventHandler<unknown>);
    return () => listeners.get(event)?.delete(handler as EventHandler<unknown>);
  },
};

// TailorWorkflow — емітує подію, не знає хто підпишеться
async function runTailor() {
  const result = await tailoringService.tailor(...);
  appEvents.emit("tailor:completed", { result });  // ← тільки це
}

// Pipeline — підписується на подію, не знає звідки вона
function usePipelineData() {
  const store = usePipelineStore();
  useEffect(() => {
    return appEvents.on("tailor:completed", ({ result }) => {
      store.setApps(prev => [...prev, result.application]);
    });
  }, []);
}
```

Тепер зміна в TailorWorkflow не ламає Pipeline і навпаки.

## Метрики складності модуля

Як зрозуміти що модуль «перевантажений» і треба ділити:

| Метрика | Норма | Тривожно |
|---------|-------|---------|
| Кількість рядків | < 200 | > 400 |
| Кількість `import`-ів | < 8 | > 15 |
| Кількість `useState` | < 5 | > 8 |
| Кількість `useEffect` | < 3 | > 5 |
| Кількість exported символів | < 3 | > 7 |

### cv-tailor зараз

| Файл | Рядки | useState | useEffect | Висновок |
|------|-------|----------|-----------|---------|
| `App.tsx` | 709 | 15+ | 5 | ❌ треба ділити |
| `ApplicationDetailPanel.tsx` | 486 | 6 | 4 | ⚠️ межа |
| `Pipeline.tsx` | 270 | 4 | 0 | ✅ ок |
| `PdfImportModal.tsx` | 276 | 5 | 1 | ✅ ок |
