## Що таке «стан» насправді

Стан — це **дані, що змінюються в часі і впливають на вивід програми**.

Але не всі дані — стан:
- Константа `SENIORITY_TEMPLATES` в App.tsx — **не стан** (не змінюється)
- `apps: Application[]` — **стан** (змінюється при delete/patch)
- `new Date()` кожен раз новий — **не стан** (обчислення)

### Категорії стану в cv-tailor

```
Стан cv-tailor
├── UI State (тільки для UI)
│   ├── tab: "tailor" | "pipeline"
│   ├── drawerOpen: boolean
│   ├── activeSection: string
│   └── search: string (Pipeline)
│
├── Local State (один компонент)
│   ├── sortKey, sortDir (Pipeline)
│   ├── notes, notesSaving (ApplicationDetailPanel)
│   └── email, loading (FollowUpSection)
│
├── App State (спільний між компонентами)
│   ├── apps: Application[]
│   ├── master: MasterCv
│   └── result: TailorResponse
│
└── Server State (cache + sync з API)
    ├── master CV (GET /api/master-cv)
    ├── applications (GET /api/applications)
    └── application data (GET /api/applications/:id/data)
```

**Різні типи потребують різних інструментів:**

| Тип | Інструмент | Чому |
|-----|-----------|------|
| UI State | `useState` локально | Нікому більше не потрібен |
| Local State | `useState` локально | Ізольований в компоненті |
| App State | Zustand store | Спільний, синхронний |
| Server State | React Query / SWR | Кешування, refetch, stale |

## Проблеми React-екосистеми зі стейтом

### Проблема 1: useState підіймається вгору (prop drilling)

Коли двом компонентам потрібен один стан — його «підіймають» до спільного батька.
В cv-tailor це призвело до `App.tsx` з 20+ state-змінних.

```
До підйому:          Після підйому:
Pipeline             App.tsx          ← стан тут (далеко)
 └── [apps]           └── Pipeline ← отримує через props
                           └── PipelineRow ← отримує через props
                                └── EditCell ← отримує через props
```

React Query і Zustand вирішують це: компонент читає прямо зі store/cache без props.

### Проблема 2: useEffect — антипатерн для синхронізації

```tsx
// Типовий антипатерн: синхронізація через useEffect
useEffect(() => {
  if (activeSection === "ats" && !atsResult) {
    runAtsCheck();
  }
}, [activeSection, atsResult]);  // ← залежності легко забути або зробити зайвими
```

`useEffect` — це «escape hatch» (клапан безпеки), не основний спосіб роботи.
**Правило**: якщо можна обчислити значення — обчислюй, не синхронізуй через effect.

### Проблема 3: Stale closure

```tsx
function CounterBug() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => {
      setCount(count + 1);  // ❌ count "заморожений" в момент створення effect
    }, 1000);
    return () => clearInterval(id);
  }, []);  // ← порожній масив залежностей = стале замикання

  return <div>{count}</div>;  // Завжди показує 1
}

// Правильно:
setCount(prev => prev + 1);  // ← функціональне оновлення, не залежить від closure
```

В cv-tailor `App.tsx` рядки 163–173: таймер `elapsed` — ось де ця проблема:
```tsx
timerRef.current = setInterval(() => setElapsed((s) => s + 1), 1000);
// ✅ Тут правильно — використовується функціональне оновлення
```

## Чому бізнес-логіка не повинна жити в UI

**Бізнес-логіка** — правила, що визначають **як програма поводиться**, незалежно від відображення.

Приклад: «заявку не можна видалити якщо статус Offer» — це бізнес-правило.

```tsx
// ❌ Бізнес-правило в компоненті
function PipelineRow({ app, onDelete }: Props) {
  function handleDelete() {
    if (app.status === "Offer") {  // ← бізнес-правило тут
      showToast("Cannot delete Offer", "error");
      return;
    }
    onDelete(app.id);
  }
  return <button onClick={handleDelete}>Delete</button>;
}

// Проблема: правило дублюється якщо є ще одне місце де можна видалити
// Тест: треба рендерити компонент щоб перевірити правило
```

```tsx
// ✅ Бізнес-правило в сервісі
// client/src/services/ApplicationService.ts
class ApplicationService {
  canDelete(app: Application): { allowed: boolean; reason?: string } {
    if (app.status === "Offer") {
      return { allowed: false, reason: "Cannot delete Offer application" };
    }
    return { allowed: true };
  }

  async delete(id: string, service: ApiService): Promise<void> {
    const app = await service.getApplication(id);
    const check = this.canDelete(app);
    if (!check.allowed) throw new Error(check.reason);
    await service.deleteApplication(id);
  }
}

// Компонент — тільки UI
function PipelineRow({ app, onDelete }: Props) {
  return (
    <button onClick={() => onDelete(app.id)}>Delete</button>
    // Помилку покаже onDelete якщо не можна видалити
  );
}

// Тест — без рендерингу:
test("cannot delete Offer", () => {
  const service = new ApplicationService();
  const result = service.canDelete({ status: "Offer" });
  expect(result.allowed).toBe(false);
});
```

## Порівняння підходів до Server State

### useState + useEffect (поточний cv-tailor)

```tsx
const [apps, setApps] = useState<Application[]>([]);
const [loading, setLoading] = useState(true);
const [error, setError] = useState<string | null>(null);

useEffect(() => {
  setLoading(true);
  api.getApplications()
    .then(setApps)
    .catch(e => setError(e.message))
    .finally(() => setLoading(false));
}, []);
```

❌ Немає кешування (кожен mount = новий fetch)
❌ Немає автоматичного refetch
❌ Бойлерплейт в кожному компоненті

### React Query (рекомендований для Server State)

```tsx
import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query";

// Завантаження — автоматичний кеш, refetch при фокусі вкладки
function usePipelineQuery() {
  return useQuery({
    queryKey: ["applications"],
    queryFn: () => api.getApplications(),
    staleTime: 30_000,  // кеш свіжий 30с
  });
}

// Мутація з автоматичним інвалідуванням кешу
function useDeleteApplication() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: (id: string) => api.deleteApplication(id),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["applications"] });
    },
  });
}

// Компонент — чисто читає стан
function Pipeline() {
  const { data: apps = [], isLoading, error } = usePipelineQuery();
  const deleteMutation = useDeleteApplication();

  return (
    <table>
      {apps.map(app => (
        <tr key={app.id}>
          <td>{app.company}</td>
          <td>
            <button onClick={() => deleteMutation.mutate(app.id)}>
              {deleteMutation.isPending ? "..." : "Delete"}
            </button>
          </td>
        </tr>
      ))}
    </table>
  );
}
```

✅ Кешування без бойлерплейту
✅ Loading/error стан автоматичний
✅ Оновлення UI після мутації
