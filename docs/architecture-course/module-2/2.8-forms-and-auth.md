# 2.8 — Робота з формами та аутентифікація

Форми — місце де архітектура найчастіше розвалюється.
Тут найбільше антипатернів, найбільше «тимчасового» коду що залишається назавжди.

---

## Головна помилка при роботі з формами

**Помилка**: змішувати **логіку форми** (валідація, submit, стан полів) з **бізнес-логікою** (що робити з даними) і **UI** (як виглядає форма).

```tsx
// ❌ Все в одному компоненті — cv-tailor зараз
// App.tsx: форма tailoring розмазана по всьому файлу

// Стан форми (рядки 100-114):
const { draft, setDraft } = useDraftAutoSave();

// Логіка submit (рядки 180-280):
async function runTailor() {
  if (!draft.jobText.trim()) return;
  // ... 100 рядків логіки
}

// UI форми (рядки 350-500):
<textarea value={draft.jobText} onChange={e => setDraft({ jobText: e.target.value })} />
<button onClick={runTailor}>Tailor</button>
```

**Правильно**: три окремі шари:
```
FormLogic (React Hook Form) ← валідація, стан полів, submit handler
      ↓
BusinessLogic (useTailorWorkflow) ← що робити з валідними даними
      ↓
UI (TailorForm.tsx) ← тільки розмітка
```

## React Hook Form: базові концепції

RHF контролює форму через **нативний DOM** (ref), а не через React state — тому немає зайвих ре-рендерів на кожне натискання клавіші.

### Порівняння: controlled vs RHF

```tsx
// ❌ Controlled input — ре-рендер на кожен символ
function TailorFormBad() {
  const [jobText, setJobText] = useState("");
  const [jobUrl, setJobUrl] = useState("");

  return (
    <form>
      <textarea
        value={jobText}
        onChange={e => setJobText(e.target.value)}  // ← ре-рендер
      />
      <input
        value={jobUrl}
        onChange={e => setJobUrl(e.target.value)}   // ← ре-рендер
      />
    </form>
  );
}

// ✅ RHF — ре-рендер тільки при валідації або submit
import { useForm } from "react-hook-form";

interface TailorFormData {
  jobText: string;
  jobUrl?: string;
  customInstructions?: string;
}

function TailorForm({ onSubmit }: { onSubmit: (data: TailorFormData) => void }) {
  const { register, handleSubmit, formState: { errors, isSubmitting } } = useForm<TailorFormData>();

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <textarea
        {...register("jobText", { required: "Paste the job description" })}
      />
      {errors.jobText && <span className="form-error">{errors.jobText.message}</span>}

      <input
        {...register("jobUrl", {
          pattern: {
            value: /^https?:\/\/.+/,
            message: "Must be a valid URL"
          }
        })}
        placeholder="Job URL (optional)"
      />

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? "Tailoring…" : "Tailor CV"}
      </button>
    </form>
  );
}
```

## Zod: типобезпечна валідація

Zod дозволяє визначити схему **один раз** — і отримати і TypeScript-тип, і runtime-валідацію.

```ts
import { z } from "zod";

// Схема = джерело правди для типу і валідації
const tailorFormSchema = z.object({
  jobText: z.string()
    .min(50, "Job description too short — paste the full text")
    .max(10_000, "Job description too long"),
  jobUrl: z.string()
    .url("Must be a valid URL starting with http(s)://")
    .optional()
    .or(z.literal("")),
  customInstructions: z.string().max(1000).optional(),
  userLevel: z.enum(["junior", "middle", "senior"]).default("middle"),
});

// Тип виводиться автоматично — не дублюємо!
type TailorFormData = z.infer<typeof tailorFormSchema>;

// Інтеграція Zod + RHF через zodResolver
import { zodResolver } from "@hookform/resolvers/zod";

const form = useForm<TailorFormData>({
  resolver: zodResolver(tailorFormSchema),
  defaultValues: { userLevel: "middle" },
});
```

### Складні схеми Zod

```ts
// Умовна валідація: або jobText, або jobUrl — одне обов'язкове
const tailorSchema = z.object({
  jobText: z.string().optional(),
  jobUrl: z.string().url().optional(),
}).refine(
  (data) => data.jobText || data.jobUrl,
  {
    message: "Provide either job text or job URL",
    path: ["jobText"],  // де показати помилку
  }
);

// Трансформація вхідних даних
const schema = z.object({
  salary: z.string()
    .transform(s => parseInt(s, 10))
    .refine(n => !isNaN(n), "Must be a number"),
});
```

## Розділення логіки форми і UI

### Принцип

```
UI компонент:          TailorFormFields.tsx
  - тільки JSX
  - register/errors з RHF через props

Form controller:       TailorFormContainer.tsx або useTailorForm.ts
  - useForm, zodResolver
  - handleSubmit
  - передає register/errors в UI

Business logic:        useTailorWorkflow.ts
  - API-виклики
  - стан процесу (stage, progress)
  - не знає про RHF
```

```tsx
// 1. Чисто UI — тільки JSX
interface TailorFieldsProps {
  register: UseFormRegister<TailorFormData>;
  errors: FieldErrors<TailorFormData>;
  isSubmitting: boolean;
}

export function TailorFormFields({ register, errors, isSubmitting }: TailorFieldsProps) {
  return (
    <>
      <textarea
        {...register("jobText")}
        className={errors.jobText ? "input-error" : ""}
        placeholder="Paste the job description..."
      />
      {errors.jobText && <p className="form-error">{errors.jobText.message}</p>}

      <input {...register("jobUrl")} placeholder="Job URL (optional)" />

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? "Tailoring…" : "Tailor CV"}
      </button>
    </>
  );
}

// 2. Form controller — знає про RHF і workflow
export function TailorForm() {
  const workflow = useTailorWorkflow();
  const { register, handleSubmit, formState: { errors, isSubmitting } } = useForm<TailorFormData>({
    resolver: zodResolver(tailorFormSchema),
  });

  return (
    <form onSubmit={handleSubmit(workflow.runTailor)}>
      <TailorFormFields register={register} errors={errors} isSubmitting={isSubmitting} />
    </form>
  );
}

// 3. Business logic — не знає про форму
function useTailorWorkflow() {
  const store = useTailorStore();
  const service = useTailoringService();

  async function runTailor(data: TailorFormData) {
    // Отримує вже валідовані дані від RHF
    store.setStage("analyzing");
    // ...
  }

  return { runTailor, stage: store.stage };
}
```

## JWT Auth: фронтенд-перспектива

JWT Auth зараз відсутній в cv-tailor — але ось як він додається **без порушення архітектури**.

### Як JWT auth вписується в Service Layer

```ts
// AuthService — свій Bounded Context
interface AuthService {
  login(credentials: LoginCredentials): Promise<void>;
  logout(): Promise<void>;
  getToken(): string | null;
  isAuthenticated(): boolean;
  refreshToken(): Promise<void>;
}

// ApiClient з автоматичним токеном (Axios interceptor або fetch wrapper)
function createApiClient(authService: AuthService): ApiClient {
  return {
    async get<T>(url: string): Promise<T> {
      const token = authService.getToken();
      const res = await fetch(url, {
        headers: token ? { Authorization: `Bearer ${token}` } : {},
      });

      // Якщо 401 — спробуй refresh
      if (res.status === 401) {
        await authService.refreshToken();
        // Повтор запиту з новим токеном
        const token = authService.getToken();
        return fetch(url, {
          headers: { Authorization: `Bearer ${token!}` },
        }).then(r => r.json());
      }

      return r.json();
    },
  };
}
```

### Protected Route без прив'язки до роутера

```tsx
// Компонент Can (з теми 2.6) + AuthGuard
interface AuthGuardProps {
  children: ReactNode;
  fallback?: ReactNode;
}

function AuthGuard({ children, fallback = <LoginPage /> }: AuthGuardProps) {
  const auth = useAuthService();
  return auth.isAuthenticated() ? <>{children}</> : <>{fallback}</>;
}

// Використання
<AuthGuard>
  <App />
</AuthGuard>
```
