## Feature vs Widget: де різниця

FSD розрізняє два рівні складних компонентів:

| | Feature | Widget |
|-|---------|--------|
| **Містить** | Бізнес-логіку, state management | Кілька features разом |
| **Self-contained** | Так | Так |
| **Залежить від** | Entities, Shared | Features, Entities, Shared |
| **Приклад** | `TailorWorkflow`, `PipelineFilter` | `ApplicationDetailPanel`, `MasterCvDrawer` |

### cv-tailor через цю лінзу

```
Widgets (великі, self-contained секції):
  ApplicationDetailPanel ← містить 6 features всередині
  MasterCvDrawer         ← містить profile management + editor

Features (ізольовані фічі з логікою):
  TailorWorkflow    ← аналіз + tailoring + прогрес
  PipelineFilter    ← пошук + фільтрація + сортування
  AtsCheck          ← перевірка ATS
  MockInterview     ← чат-інтерфейс
  FollowUpDraft     ← генерація follow-up
  CoverLetter       ← генерація cover letter
```

## Page-first підхід у проектуванні

**Помилка**: проектувати компоненти знизу вгору (спочатку атоми, потім молекули, потім сторінка).

**Правило**: починай з **сторінки** — визнач layout, зони, інформацію. Потім ділиш на компоненти.

### Page-first для Pipeline

```
Крок 1: Намалюй сторінку
┌──────────────────────────────────────────────────┐
│  [Stats: Total | Applied | Interviewing | Offers] │
├──────────────────────────────────────────────────┤
│  [Search...]  [Status: All ▼]  [Sort: Date ▼]   │
├──────────────────────────────────────────────────┤
│  Company │ Role │ Status │ Fit │ Date │ Actions  │
│  ─────────────────────────────────────────────  │
│  Acme    │ SWE  │ Applied│ 82% │ 3d   │ [Open]  │
│  ...                                            │
└──────────────────────────────────────────────────┘

Крок 2: Визнач зони відповідальності
- PipelineStats     ← верхня панель (read-only)
- PipelineControls  ← фільтри (local state)
- PipelineTable     ← таблиця (читає з store)
- PipelineRow       ← один рядок (callbacks up)

Крок 3: Визнач потоки даних
- PipelineStats ← apps[] (read-only)
- PipelineControls → фільтри (local state) → передає в PipelineTable
- PipelineTable ← filteredApps (computed)
- PipelineRow → onPatch, onDelete, onOpen (callbacks)
```

## SSoT — Single Source of Truth

**SSoT**: кожне значення існує в **одному місці**. Решта — обчислення з нього.

### Антипатерн: дублювання стану

```tsx
// ❌ filteredApps збережений в state — двічі!
const [apps, setApps] = useState<Application[]>([]);
const [filteredApps, setFilteredApps] = useState<Application[]>([]);  // ← копія!

// При зміні apps треба вручну оновити filteredApps
useEffect(() => {
  setFilteredApps(apps.filter(/* ... */));
}, [apps, filter]);
// Легко забути оновити, стани розходяться
```

```tsx
// ✅ filteredApps — обчислення, не стан
const [apps, setApps] = useState<Application[]>([]);  // SSoT
const [filter, setFilter] = useState<FilterState>({ search: "", status: "All" });

// filteredApps обчислюється, не зберігається
const filteredApps = useMemo(() =>
  apps.filter(a => {
    const matchSearch = !filter.search || a.company.toLowerCase().includes(filter.search);
    const matchStatus = filter.status === "All" || a.status === filter.status;
    return matchSearch && matchStatus;
  }),
  [apps, filter]  // автоматично оновлюється
);
```

### SSoT в cv-tailor: Pipeline вже правильно!

```tsx
// client/src/components/Pipeline.tsx рядки 35-53
const visibleApplications = useMemo(() => {
  // ✅ SSoT: apps — єдине джерело, visibleApplications — обчислення
  const q = search.trim().toLowerCase();
  const filtered = applications.filter((a) => { /* ... */ });
  return sortKey ? [...filtered].sort(/* ... */) : filtered;
}, [applications, search, sortDir, sortKey, statusFilter]);
```

**Це правильний патерн** — `visibleApplications` обчислюється, не зберігається.

## Нормалізація даних

При складних відносинах між сутностями — нормалізуй дані як в базі даних.

### Денормалізований стан (поточний підхід)

```ts
// Масив об'єктів — пошук за id = O(n)
const apps: Application[] = [
  { id: "app_1", company: "Acme", status: "Applied" },
  { id: "app_2", company: "Globex", status: "Interview" },
];

// Пошук: apps.find(a => a.id === id)  — O(n)
// Оновлення: apps.map(a => a.id === id ? {...a, ...patch} : a)  — O(n)
```

### Нормалізований стан (масштабується)

```ts
// id → об'єкт: пошук = O(1)
interface NormalizedPipeline {
  byId: Record<string, Application>;
  ids: string[];  // порядок
}

const pipeline: NormalizedPipeline = {
  byId: {
    "app_1": { id: "app_1", company: "Acme", status: "Applied" },
    "app_2": { id: "app_2", company: "Globex", status: "Interview" },
  },
  ids: ["app_1", "app_2"],
};

// Пошук: pipeline.byId[id]  — O(1)
// Оновлення: { ...pipeline, byId: { ...pipeline.byId, [id]: {...pipeline.byId[id], ...patch} } }

// Для рендерингу:
const apps = pipeline.ids.map(id => pipeline.byId[id]);
```

**Коли нормалізація потрібна**: > 100 записів, часті пошуки за id, вкладені відносини.
В cv-tailor Pipeline — поки достатньо масиву.

## Undo/Redo через Event Sourcing

**Event Sourcing**: зберігаємо не поточний стан, а **список подій** що його породили.
Undo = видалення останньої події. Redo = повтор видаленої.

### Простий Undo для Pipeline delete

```ts
// client/src/hooks/useUndoableDelete.ts

interface DeleteEvent {
  type: "DELETE";
  app: Application;
  timestamp: number;
}

export function useUndoableDelete(
  onDelete: (id: string) => void,
  onRestore: (app: Application) => void
) {
  const [history, setHistory] = useState<DeleteEvent[]>([]);

  function deleteWithUndo(app: Application, undoWindowMs = 5000) {
    // 1. Записуємо подію
    const event: DeleteEvent = { type: "DELETE", app, timestamp: Date.now() };
    setHistory(prev => [...prev, event]);

    // 2. Виконуємо видалення
    onDelete(app.id);

    // 3. Показуємо undo-кнопку
    showToast(`Removed ${app.company}`, "info", {
      action: {
        label: "Undo",
        onClick: () => {
          // Undo: відновлюємо
          onRestore(app);
          setHistory(prev => prev.filter(e => e.timestamp !== event.timestamp));
        },
      },
      duration: undoWindowMs,
    });
  }

  return { deleteWithUndo, history };
}
```

### Повноцінний Undo/Redo стек

```ts
// Для складних операцій: стек команд
interface Command {
  execute(): void;
  undo(): void;
}

function useCommandHistory() {
  const [past, setPast] = useState<Command[]>([]);
  const [future, setFuture] = useState<Command[]>([]);

  function execute(command: Command) {
    command.execute();
    setPast(prev => [...prev, command]);
    setFuture([]);  // нова дія очищає redo-стек
  }

  function undo() {
    if (!past.length) return;
    const command = past[past.length - 1];
    command.undo();
    setPast(prev => prev.slice(0, -1));
    setFuture(prev => [command, ...prev]);
  }

  function redo() {
    if (!future.length) return;
    const command = future[0];
    command.execute();
    setFuture(prev => prev.slice(1));
    setPast(prev => [...prev, command]);
  }

  return { execute, undo, redo, canUndo: past.length > 0, canRedo: future.length > 0 };
}

// Команда видалення заявки
function createDeleteCommand(
  app: Application,
  store: PipelineStore,
  service: ApplicationService
): Command {
  return {
    execute() {
      store.remove(app.id);
      service.deleteApplication(app.id).catch(/* rollback */);
    },
    undo() {
      store.addApp(app);
      service.createApplication(app).catch(/* rollback */);
    },
  };
}
```

## Local-first архітектура (концепція)

**Local-first**: дані зберігаються на клієнті, синхронізуються з сервером у фоні.
Прикладом є Linear — він працює без інтернету, зміни синхронізуються коли є зв'язок.

### Чому це важливо для cv-tailor

Зараз:
- Форма tailoring втрачається при закритті вкладки (частково вирішено через `useDraftAutoSave`)
- Pipeline не працює офлайн
- Кожна дія чекає API відповідь

Local-first підхід:
```
Дія → localStorage/IndexedDB → UI оновлюється ОДРАЗУ
                             → sync queue → API у фоні
```

### Спрощений local-first для Pipeline

```ts
// client/src/services/LocalFirstApplicationService.ts

export function createLocalFirstService(
  localStore: LocalStore,  // localStorage/IndexedDB
  apiService: ApplicationService,
): ApplicationService {
  return {
    async getApplications() {
      // Повертаємо з кешу одразу
      const cached = localStore.getApplications();
      if (cached) {
        // Фоновий refetch
        apiService.getApplications().then(fresh => {
          localStore.setApplications(fresh);
        }).catch(console.error);
        return cached;
      }
      // Якщо кешу немає — чекаємо API
      const data = await apiService.getApplications();
      localStore.setApplications(data);
      return data;
    },

    async patchApplication(id, patch) {
      // 1. Одразу оновлюємо локально
      localStore.patchApplication(id, patch);
      // 2. Синхронізуємо з API у фоні
      return apiService.patchApplication(id, patch);
    },
  };
}
```
