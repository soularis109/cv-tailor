## Чому папки — це архітектурне рішення

Структура папок — це перша документація проєкту. Вона відповідає на питання:
«З чого складається ця програма?»

Є два популярних підходи:

### Організація за типом файлу (type-based)
```
src/
  components/    ← всі компоненти разом
  hooks/         ← всі хуки разом
  utils/         ← всі утиліти разом
```
**Проблема**: коли зростає — незрозуміло що з чим пов'язано.
`api.ts` і `Pipeline.tsx` лежать в одному рівні, хоча це різні шари.

### Організація за шаром (layer-based) ← рекомендований для cv-tailor
```
src/
  services/    ← інфраструктура і бізнес-логіка
  stores/      ← стан застосунку
  hooks/       ← ViewModel (хуки що комбінують stores + services)
  components/  ← View (тільки рендер)
  types/       ← Model (типи і структури даних)
  utils/       ← чисті функції без залежностей
```

## П'ять архітектурних шарів

```
┌─────────────────────────────────────────────────────┐
│                      View                           │
│   components/  (тільки JSX + хуки)                 │
└──────────────────────────┬──────────────────────────┘
                           │ викликає
┌──────────────────────────▼──────────────────────────┐
│                    ViewModel                        │
│   hooks/  (комбінують Store + Service)              │
└──────────────────────────┬──────────────────────────┘
                           │ читає/пише
┌──────────────────────────▼──────────────────────────┐
│                     Store                           │
│   stores/  (глобальний реактивний стан)             │
└──────────────────────────┬──────────────────────────┘
                           │ викликає
┌──────────────────────────▼──────────────────────────┐
│                    Service                          │
│   services/  (бізнес-логіка, API, інфраструктура)   │
└──────────────────────────┬──────────────────────────┘
                           │ повертає
┌──────────────────────────▼──────────────────────────┐
│                     Model                           │
│   types/   (структури даних, Zod схеми)             │
└─────────────────────────────────────────────────────┘
```

**Правило напрямку залежностей**: стрілки йдуть **тільки вниз**.
- View знає про ViewModel
- ViewModel знає про Store і Service
- Store знає про Model
- Service знає про Model
- **Нижній шар не знає про верхній**

## Інфраструктурний код

Інфраструктурний код — це все що **не є бізнес-логікою**, але без чого програма не працює:
- HTTP-клієнт (`api.ts`)
- LocalStorage адаптер
- Звуки/нотифікації (`notify.ts`)
- Таймери

**Правило**: інфраструктурний код живе в `services/` або `utils/` і **завжди замотаний в інтерфейс** (DIP з теми 1.1).

### Поточна ситуація cv-tailor

```
src/utils/notify.ts  ← інфраструктура прямо в utils (ок)
src/utils/toast.ts   ← pub/sub реалізація (інфраструктура)
src/api.ts           ← HTTP без інтерфейсу (проблема — немає шару)
```

### Ціль

```
src/services/
  api/
    ApplicationService.ts      ← інтерфейс
    ApiApplicationService.ts   ← fetch-реалізація
    MockApplicationService.ts  ← mock для розробки
  notification/
    NotificationService.ts     ← інтерфейс
    BrowserNotificationService.ts ← Web Notifications API
```

## Contract-first API

**Contract-first** — спочатку визначаєш **контракт** (інтерфейс), потім реалізацію.

Переваги:
- Frontend і Backend можуть розроблятись паралельно
- Заміна реалізації без зміни компонентів
- Mock = просто об'єкт що реалізує контракт

### Приклад для cv-tailor

```ts
// Крок 1: контракт (незалежно від того чи бекенд готовий)
export interface TailoringService {
  analyzeJob(jobText: string, jobUrl?: string): Promise<JobAnalysis>;
  tailorCv(
    jobText: string,
    cv: MasterCv,
    options: TailoringOptions
  ): Promise<TailorResponse>;
}

// Крок 2: mock для розробки (frontend не чекає на бекенд)
export const mockTailoringService: TailoringService = {
  async analyzeJob() {
    await delay(1000);  // симуляція мережі
    return { role_title: "Senior Frontend", must_have: ["React", "TypeScript"], /* ... */ };
  },
  async tailorCv() {
    await delay(2000);
    return { tailored: mockCv, analysis: mockAnalysis };
  },
};

// Крок 3: реалізація через fetch (коли бекенд готовий)
export const apiTailoringService: TailoringService = {
  async analyzeJob(jobText, jobUrl) {
    return json(await fetch("/api/analyze", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ jobText, jobUrl }),
    }));
  },
  // ...
};
```

## ДЗ 8 — Реструктуризація папок cv-tailor

### Завдання

Створи нову структуру папок (без переносу коду — тільки директорії і порожні файли).
Мета: побачити архітектуру **до** того як починаєш рефакторинг.

1. Створи директорії: `types/`, `services/`, `stores/`, `components/ui/`
2. Створи порожні файли `index.ts` в кожній директорії
3. В `types/index.ts` напиши коментарі: що буде re-exported звідки
4. В `services/contexts.tsx` напиши коментар: які сервіси тут будуть

### Шаблон

```bash
# Запусти в client/src/
mkdir -p types services stores components/ui

# Створи index файли
touch types/index.ts services/contexts.tsx stores/.gitkeep
```
