# 2.2 — Стейт-менеджер і його роль в архітектурі

**Головна ідея**: стейт-менеджер — це **інструмент синхронізації UI і стану**.
Він не є центром архітектури. Бізнес-логіка живе в сервісах, а стейт-менеджер лише робить реакцію UI на зміни автоматичною.

---

## Що таке State і яким він буває

Не весь стан однаковий. Розрізняй чотири типи:

| Тип | Опис | Де живе | Приклад в cv-tailor |
|-----|------|---------|---------------------|
| **UI State** | Вигляд інтерфейсу | `useState` в компоненті | `drawerOpen`, `activeSection` |
| **Local State** | Стан одного компонента, не потрібен нікому | `useState` в компоненті | `search`, `sortKey` в Pipeline |
| **App State** | Спільний між компонентами, синхронний | Zustand store | `apps`, `tailorResult` |
| **Server State** | Дані з API, потребують кешування | React Query / SWR | `masterCv`, `applications` |

**Помилка новачка**: кидати всі 4 типи в один глобальний store.
**Правило**: кожен тип стану — своє місце.

---

## Поточна ситуація: все в App.tsx

```tsx
// client/src/App.tsx — 20+ useState

// UI State (правильно тут)
const [tab, setTab] = useState<Tab>("tailor");
const [drawerOpen, setDrawerOpen] = useState(false);

// App State (неправильно тут — треба store)
const [apps, setApps] = useState<Application[]>([]);
const [result, setResult] = useState<TailorResponse | null>(null);

// Server State (неправильно тут — треба React Query або store з fetch)
const [master, setMaster] = useState<MasterCv | null>(null);
const [masterLoading, setMasterLoading] = useState(true);
```

**Проблема**: `App.tsx` є єдиним джерелом правди для *всього*.
Будь-який компонент що хоче дані — повинен отримати їх через prop drilling з App.

## Zustand: правильне використання

Zustand — простий стейт-менеджер без бойлерплейту. Але є нюанс:

**Неправильно** — бізнес-логіка в store:
```ts
// ❌ Store знає про API, має side-effects
export const usePipelineStore = create((set) => ({
  apps: [],
  async loadApps() {
    const data = await fetch("/api/applications").then(r => r.json());
    set({ apps: data });
  },
  async deleteApp(id: string) {
    await fetch(`/api/applications/${id}`, { method: "DELETE" });
    set(state => ({ apps: state.apps.filter(a => a.id !== id) }));
  },
}));
```

**Правильно** — store тільки зберігає стан, логіка в сервісі:
```ts
// ✅ Store = реактивне сховище (тільки get/set)
interface PipelineState {
  apps: Application[];
  loading: boolean;
  error: string | null;
  setApps: (apps: Application[]) => void;
  setLoading: (v: boolean) => void;
  setError: (e: string | null) => void;
  patch: (id: string, update: Partial<Application>) => void;
  remove: (id: string) => void;
}

export const usePipelineStore = create<PipelineState>((set) => ({
  apps: [],
  loading: false,
  error: null,
  setApps: (apps) => set({ apps }),
  setLoading: (loading) => set({ loading }),
  setError: (error) => set({ error }),
  patch: (id, update) =>
    set((state) => ({
      apps: state.apps.map((a) => (a.id === id ? { ...a, ...update } : a)),
    })),
  remove: (id) =>
    set((state) => ({ apps: state.apps.filter((a) => a.id !== id) })),
}));

// ✅ Сервіс-шар — бізнес-логіка + виклики API + store
export function usePipelineData(service: ApplicationService) {
  const store = usePipelineStore();

  useEffect(() => {
    store.setLoading(true);
    service.getApplications()
      .then(store.setApps)
      .catch((e) => store.setError(e.message))
      .finally(() => store.setLoading(false));
  }, []);

  async function deleteApp(id: string) {
    store.remove(id);  // optimistic
    try {
      await service.deleteApplication(id);
    } catch {
      store.setApps(await service.getApplications());  // rollback
    }
  }

  return { ...store, deleteApp };
}
```

## Селектори: уникай зайвих ре-рендерів

```tsx
// ❌ Весь store — компонент перерендериться при будь-якій зміні
function PipelineStats() {
  const { apps, loading, error, setApps, patch, remove } = usePipelineStore();
  return <div>{apps.length} applications</div>;
}

// ✅ Тільки потрібна частина — ре-рендер тільки коли apps.length змінився
function PipelineStats() {
  const count = usePipelineStore((state) => state.apps.length);
  return <div>{count} applications</div>;
}

// ✅ Складний селектор зі збереженням референції
function PipelineFilteredApps() {
  const apps = usePipelineStore(
    useShallow((state) => state.apps.filter((a) => a.status === "Applied"))
  );
  return <AppsTable apps={apps} />;
}
```

**Правило**: компонент підписується тільки на те що відображає.

## Global / Module / Local Composition Root

До цього ми говорили про один Composition Root в `main.tsx`.
На практиці є три рівні:

```
Global Composition Root (main.tsx)
  └── App.tsx
        ├── Module Composition Root (PipelineModule.tsx)
        │     ├── PipelineServiceContext.Provider
        │     ├── usePipelineStore
        │     └── Pipeline компоненти
        │
        └── Module Composition Root (TailorModule.tsx)
              ├── TailoringServiceContext.Provider
              └── Tailor компоненти
```

| Рівень | Де | Що вирішує |
|--------|-----|-----------|
| **Global** | `main.tsx` | Глобальні сервіси (auth, notifications) |
| **Module** | `XxxFeature.tsx` або `XxxProvider.tsx` | Сервіси конкретної фічі |
| **Local** | Всередині компонента | `useState` для UI-стану цього компонента |

## Service Locator: коли і чому з обережністю

Service Locator — це глобальний реєстр сервісів. Замість DI через Context — централізований об'єкт:

```ts
// service-locator.ts
const services = new Map<string, unknown>();

export const ServiceLocator = {
  register<T>(key: string, service: T) {
    services.set(key, service);
  },
  get<T>(key: string): T {
    const service = services.get(key);
    if (!service) throw new Error(`Service ${key} not registered`);
    return service as T;
  },
};

// Composition Root:
ServiceLocator.register("TailoringService", apiTailoringService);

// Компонент:
const service = ServiceLocator.get<TailoringService>("TailoringService");
```

**Коли ок**: у Module Composition Root де контроль над lifecycle зрозумілий.
**Коли погано**: як заміна DI скрізь — прихований coupling, тяжко тестувати.

**Рекомендація**: в cv-tailor використовуй **Context** (tema 1.3). Service Locator — коли є конкретна причина.

## Tailored store для cv-tailor

```ts
// client/src/stores/tailorStore.ts
import { create } from "zustand";
import type { JobAnalysis, TailorResponse } from "../types";

type Stage = null | "analyzing" | "tailoring";

interface TailorState {
  stage: Stage;
  progress: number;
  elapsed: number;
  error: string | null;
  result: TailorResponse | null;
  cachedAnalysis: JobAnalysis | null;
  companyBrief: string | null;

  setStage: (s: Stage) => void;
  setProgress: (p: number) => void;
  setElapsed: (e: number) => void;
  setError: (e: string | null) => void;
  setResult: (r: TailorResponse | null) => void;
  setCachedAnalysis: (a: JobAnalysis | null) => void;
  setCompanyBrief: (b: string | null) => void;
  reset: () => void;
}

const initialState = {
  stage: null,
  progress: 0,
  elapsed: 0,
  error: null,
  result: null,
  cachedAnalysis: null,
  companyBrief: null,
};

export const useTailorStore = create<TailorState>((set) => ({
  ...initialState,
  setStage: (stage) => set({ stage }),
  setProgress: (progress) => set({ progress }),
  setElapsed: (elapsed) => set({ elapsed }),
  setError: (error) => set({ error }),
  setResult: (result) => set({ result }),
  setCachedAnalysis: (cachedAnalysis) => set({ cachedAnalysis }),
  setCompanyBrief: (companyBrief) => set({ companyBrief }),
  reset: () => set(initialState),
}));
```
