# 2.4 — Одна фіча, чотири реалізації

Одне і те саме завдання можна вирішити на різних рівнях «правильності».
Вибір рівня залежить від **дедлайну, складності, контексту**.

Розбираємо на прикладі фічі **«відображення FitScore»** в cv-tailor.

---

## Рівень 0: Швидке рішення (MVP)

> «Зробити зараз, думати потім»

Коли підходить:
- Прототип / хакатон
- Невідомо чи фіча взагалі виживе
- Дедлайн завтра

```tsx
// Все прямо в компоненті — без ізоляції
function ApplicationDetailPanel({ application }: { application: Application }) {
  // score прямо тут
  const score = application.fitScore;
  const color = score >= 75 ? "#22c55e" : score >= 50 ? "#f59e0b" : "#ef4444";
  const label = score >= 75 ? "Strong match" : score >= 50 ? "Worth a shot" : "Long shot";

  return (
    <div>
      <div style={{ color, fontSize: "2rem", fontWeight: "bold" }}>{score}%</div>
      <div style={{ color }}>{label}</div>
    </div>
  );
}
```

**Мінуси**: не можна перевикористати, логіка розкидана по компоненту, зміна правил = ручний пошук.

## Рівень 1: Чистий компонент з одним props

> «Виокремив, але без зайвого»

Коли підходить:
- Фіча виглядає стабільно
- Є 2+ місця де потрібен той самий UI
- Є час на базовий рефакторинг

```tsx
// client/src/components/FitGauge.tsx — реальний код cv-tailor
// Це Рівень 1 — компонент отримує score через props

interface FitGaugeProps {
  score: number;  // 0–100
}

export function FitGauge({ score }: FitGaugeProps) {
  // Логіка трансформації — всередині компонента (не ідеально, але краще)
  const band = score >= 75 ? "strong" : score >= 50 ? "maybe" : "low";
  const label = { strong: "Strong match", maybe: "Worth a shot", low: "Long shot" }[band];

  return (
    <div className={`fit-gauge fit-${band}`}>
      <span className="fit-score">{score}%</span>
      <span className="fit-label">{label}</span>
    </div>
  );
}

// Використання:
<FitGauge score={application.fitScore} />
<FitGauge score={data.tailored.fit_score} />
```

**FitGauge у cv-tailor вже на Рівні 1** — це хорошо.

## Рівень 2: Виокремлена бізнес-логіка + DI

> «Логіка окремо від UI, залежності інжектовані»

Коли підходить:
- Логіка трансформації стає складною (правила змінюються)
- Хочеш тестувати логіку без рендерингу
- Правила визначаються зовні (конфіг, бекенд)

```tsx
// 1. Бізнес-логіка виокремлена в чисту функцію
// client/src/utils/fitScore.ts

export type FitBand = "strong" | "maybe" | "low";

export interface FitScoreViewModel {
  score: number;
  band: FitBand;
  label: string;
  colorClass: string;
}

export function computeFitScore(score: number): FitScoreViewModel {
  const band: FitBand = score >= 75 ? "strong" : score >= 50 ? "maybe" : "low";
  return {
    score,
    band,
    label: { strong: "Strong match", maybe: "Worth a shot", low: "Long shot" }[band],
    colorClass: `fit-${band}`,
  };
}

// 2. Компонент отримує вже готовий ViewModel — не рахує нічого сам
interface FitGaugeProps {
  vm: FitScoreViewModel;
}

export function FitGauge({ vm }: FitGaugeProps) {
  return (
    <div className={`fit-gauge ${vm.colorClass}`}>
      <span className="fit-score">{vm.score}%</span>
      <span className="fit-label">{vm.label}</span>
    </div>
  );
}

// 3. Або компонент залишається простим, але хук рахує ViewModel
function useApplicationDetail(id: string, service: ApplicationService) {
  const [data, setData] = useState<ApplicationData | null>(null);

  useEffect(() => {
    service.getApplicationData(id).then(setData);
  }, [id]);

  const fitScore = data ? computeFitScore(data.tailored.fit_score) : null;

  return { data, fitScore };
}
```

**Переваги**: `computeFitScore` тестується як чиста функція:
```ts
test("strong match above 75", () => {
  expect(computeFitScore(80).band).toBe("strong");
});
```

## Рівень 3: ui-kit компонент (відв'язаний від домену)

> «Можна вирвати і перевикористати в іншому проєкті»

Коли підходить:
- Компонент використовується в 3+ місцях
- Команда планує ui-kit / design system
- Компонент не містить доменних знань (слів «Application», «fitScore» і т.д.)

```tsx
// client/src/components/ui/Gauge.tsx — generic, без cv-tailor специфіки

interface GaugeProps {
  value: number;           // 0–100
  label?: string;
  bands?: GaugeBand[];
  size?: "sm" | "md" | "lg";
}

interface GaugeBand {
  min: number;
  max: number;
  color: string;
  label: string;
}

const DEFAULT_BANDS: GaugeBand[] = [
  { min: 0,  max: 50, color: "#ef4444", label: "Low" },
  { min: 50, max: 75, color: "#f59e0b", label: "Medium" },
  { min: 75, max: 100, color: "#22c55e", label: "High" },
];

export function Gauge({ value, label, bands = DEFAULT_BANDS, size = "md" }: GaugeProps) {
  const band = bands.findLast(b => value >= b.min) ?? bands[0];
  return (
    <div className={`gauge gauge-${size}`} style={{ "--gauge-color": band.color } as React.CSSProperties}>
      <div className="gauge-value">{value}%</div>
      {label && <div className="gauge-label">{label ?? band.label}</div>}
    </div>
  );
}

// cv-tailor адаптер — доменна логіка в одному місці
function FitGauge({ score }: { score: number }) {
  const { label } = computeFitScore(score);
  return <Gauge value={score} label={label} />;
}
```

**Тепер `Gauge`** — можна копіювати в будь-який проєкт. `FitGauge` — адаптер з доменною логікою.

## Паттерни renderProp і slot для гнучкості

### Render Prop — коли логіка загальна, але UI різний

```tsx
// Загальна логіка підвантаження, але UI варіативний
interface AsyncStateProps<T> {
  data: T | null;
  loading: boolean;
  error: string | null;
  children: (data: T) => React.ReactNode;
  renderLoading?: () => React.ReactNode;
  renderError?: (error: string) => React.ReactNode;
}

function AsyncState<T>({ data, loading, error, children, renderLoading, renderError }: AsyncStateProps<T>) {
  if (loading) return renderLoading ? renderLoading() : <div>Loading…</div>;
  if (error) return renderError ? renderError(error) : <div className="error">{error}</div>;
  if (!data) return null;
  return children(data);
}

// Використання:
<AsyncState data={appData} loading={loading} error={error}>
  {(data) => <CvPreview cv={data.tailored} />}
</AsyncState>
```

### Slot Pattern — компонент з «дірками» для контенту

```tsx
// ApplicationDetailPanel з slots замість if-chain
interface DetailPanelProps {
  header: React.ReactNode;      // slot для хедера
  tabs: React.ReactNode;        // slot для табів
  body: React.ReactNode;        // slot для тіла
  downloads?: React.ReactNode;  // опціональний slot
}

function DetailPanel({ header, tabs, body, downloads }: DetailPanelProps) {
  return (
    <aside className="detail-panel">
      <div className="detail-header">{header}</div>
      {downloads && <div className="detail-downloads">{downloads}</div>}
      <div className="detail-tabs">{tabs}</div>
      <div className="detail-body">{body}</div>
    </aside>
  );
}

// Використання — панель знає про структуру, не про вміст
<DetailPanel
  header={<ApplicationHeader app={application} />}
  tabs={<SectionTabs sections={SECTIONS} active={activeSection} onChange={setActiveSection} />}
  body={<SectionBody section={activeSection} data={appData} />}
  downloads={data ? <DownloadButtons data={data} /> : null}
/>
```
