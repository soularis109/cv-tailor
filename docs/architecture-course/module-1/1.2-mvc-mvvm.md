# 1.2 — MVC / MVP / MVVM

MV*-патерни — це способи **розподілити відповідальності** між шарами коду.
Вони виникли ще до React — але розуміння їх дозволяє дивитись на будь-який фреймворк через одну архітектурну лінзу.

---

## Навіщо це взагалі потрібно?

Без шарів код виглядає так (це cv-tailor сьогодні):

```
App.tsx:
  - Зберігає дані заявок         ← це Model
  - Робить fetch до API          ← це Model/Service  
  - Обробляє кліки кнопок        ← це Controller
  - Рендерить JSX                ← це View
  - Запускає таймери             ← це Controller
  - Управляє toast-нотифікаціями ← це Service
```

Один файл = 6 ролей = хаос при зміні будь-якої з них.

MV*-патерни кажуть: **розділи ці ролі**, і кожна зміна буде ізольована.

---

## MVC — Model / View / Controller

```
┌──────────┐    команди    ┌────────────┐   запити/оновлення   ┌───────────┐
│          │ ────────────► │            │ ◄──────────────────── │           │
│   View   │               │ Controller │                        │   Model   │
│  (UI)    │ ◄──────────── │            │ ──────────────────────►│  (дані)   │
│          │   відповідь   └────────────┘    зміна стану         └───────────┘
└──────────┘
```

| Роль | Відповідає за | Приклад |
|------|--------------|---------|
| **Model** | Дані, бізнес-правила, стан | Список заявок, CV, статус |
| **View** | Відображення, реакція на дії | JSX, форма, таблиця |
| **Controller** | Обробка подій, оркестрація | `onClick`, `useEffect`, API-виклики |

**Потік**: User → View → Controller → Model → View

### MVC у cv-tailor — як має бути

```
View:        Pipeline.tsx, ApplicationDetailPanel.tsx (тільки рендер)
Controller:  usePipelineData, useTailorWorkflow, useApplicationsData (логіка)
Model:       Об'єкти Application, TailoredCv, JobAnalysis (дані + правила)
```

### MVC у cv-tailor — як є зараз

```
App.tsx = Model + Controller + View перемішані
│
├── Model:      useState для apps, master, result
├── Controller: useEffect + обробники runTailor, handleDelete
└── View:       JSX рядки 300-709
```

Всі три ролі в одному компоненті — тому будь-яка зміна ризикує зачепити всі три.

## MVP — Model / View / Presenter

```
┌──────────┐    події     ┌───────────┐   запити    ┌───────────┐
│          │ ────────────►│           │ ────────────►│           │
│   View   │              │ Presenter │              │   Model   │
│  (UI)    │ ◄──────────── │           │ ◄────────── │  (дані)   │
│          │   оновлення  └───────────┘  результат  └───────────┘
└──────────┘
```

Різниця від MVC:
- **View знає тільки про Presenter** — не знає про Model
- **Presenter не знає про UI** — тільки про чисту логіку
- Все це робить Presenter легко **тестованим** без рендерингу

### MVP у React — через кастомний хук

```tsx
// Presenter = хук, що не знає про JSX
function usePipelinePresenter() {
  const [apps, setApps] = useState<Application[]>([]);
  
  // Бізнес-правило: не можна видалити заявку зі статусом "Offer"
  function deleteApp(id: string) {
    const app = apps.find(a => a.id === id);
    if (app?.status === "Offer") {
      showToast("Cannot delete an Offer application", "error");
      return;
    }
    setApps(prev => prev.filter(a => a.id !== id));
    api.deleteApplication(id);
  }

  return { apps, deleteApp };
}

// View = компонент що використовує Presenter і не знає про бізнес-правила
function PipelineView() {
  const { apps, deleteApp } = usePipelinePresenter();
  
  return (
    <table>
      {apps.map(app => (
        <tr key={app.id}>
          <td>{app.company}</td>
          <td><button onClick={() => deleteApp(app.id)}>Delete</button></td>
        </tr>
      ))}
    </table>
  );
}
```

Тепер `usePipelinePresenter` тестується без рендерингу: просто викликаєш функції і перевіряєш стан.

## MVVM — Model / View / ViewModel

```
┌──────────┐  ◄──── two-way binding ────►  ┌───────────────┐  ◄──►  ┌───────────┐
│          │                               │               │         │           │
│   View   │                               │   ViewModel   │         │   Model   │
│  (UI)    │                               │  (реактивний) │         │  (дані)   │
│          │                               │               │         │           │
└──────────┘                               └───────────────┘         └───────────┘
```

Ключова ідея: **ViewModel** — це реактивне дзеркало Model для View.
- View не знає про Model — тільки про ViewModel
- ViewModel трансформує сирі дані Model у формат зручний для відображення
- **Two-way binding**: зміна у View автоматично оновлює ViewModel (і навпаки)

### MVVM у React: як реалізується реактивність

React реалізує «half-MVVM»: state — це ViewModel, `useState`/`zustand` — механізм binding.

```tsx
// Model — сирі дані
interface Application {
  fitScore: number;        // 0-100
  status: Status;
  dateAdded: string;       // ISO string "2024-01-15"
}

// ViewModel — трансформовані дані для View
// (те що хук повертає = ViewModel)
function usePipelineViewModel(apps: Application[]) {
  return apps.map(app => ({
    ...app,
    // Трансформації для UI:
    fitLabel: app.fitScore >= 75 ? "Strong" : app.fitScore >= 50 ? "Maybe" : "Long shot",
    fitClass: app.fitScore >= 75 ? "fit-strong" : app.fitScore >= 50 ? "fit-maybe" : "fit-low",
    daysAgo: daysSince(app.dateAdded),
    needsFollowUp: app.status === "Applied" && daysSince(app.dateAdded) >= 7,
  }));
}

// View — рендерить ViewModel, не знає про сирі числа
function PipelineRow({ app }: { app: ReturnType<typeof usePipelineViewModel>[0] }) {
  return (
    <tr className={app.needsFollowUp ? "row-followup" : ""}>
      <td><span className={app.fitClass}>{app.fitLabel}</span></td>
      <td>{app.daysAgo} days ago</td>
    </tr>
  );
}
```

### Data-binding через Proxy: як React насправді працює

Звичайний об'єкт не «знає» що його хтось читає. Реактивність в React — не через Proxy (на відміну від Vue/MobX), а через **ручне оголошення залежностей** (`useState`, `useEffect`).

```tsx
// React-підхід (pull-based): ти кажеш "залежу від цього state"
const [count, setCount] = useState(0);  // View перерендерить коли count зміниться

// Vue/MobX-підхід (push-based через Proxy): автоматичне відстеження
const state = reactive({ count: 0 });  // Vue сам відстежує хто читає state.count
```

Коли в React кажуть «Zustand — це ViewModel» — мають на увазі що Zustand store:
1. Зберігає стан (Model-layer)
2. Виставляє методи для мутацій (команди)
3. Повідомляє View через `useSyncExternalStore` (реактивний binding)

## Карта ролей cv-tailor: зараз vs ціль

### Зараз (все змішано)

```
App.tsx
├── [Model]      useState: apps, master, result, stage
├── [Controller] useEffect: fetch apps, fetch master, timer
├── [Controller] handlers: runTailor, handleDelete, handlePatch
└── [View]       JSX: форма tailoring + header + footer

ApplicationDetailPanel.tsx
├── [Model]      useState: data, notes, atsResult
├── [Controller] useEffect: api.getApplicationData, auto-ats
├── [Controller] handlers: saveNotes, handleRunAtsCheck
└── [View]       JSX: 6 секцій
```

### Ціль (після Модуля 1)

```
Model (дані та правила):
  types.ts — структури Application, TailoredCv, JobAnalysis

Controller/ViewModel (логіка):
  hooks/useTailorWorkflow.ts    — процес аналізу + tailoring
  hooks/usePipelineData.ts      — CRUD заявок
  hooks/useApplicationDetail.ts — деталі однієї заявки

View (тільки рендер):
  components/Pipeline.tsx           — таблиця заявок
  components/ApplicationDetailPanel.tsx — бічна панель
  components/TailorForm.tsx         — форма tailoring

Service (інфраструктура):
  services/ApplicationService.ts    — інтерфейс
  services/ApiApplicationService.ts — реалізація через fetch
```

## ДЗ 6 — Намалювати MVVM-схему для TailorWorkflow

### Завдання

Намалюй ASCII-схему архітектури `useTailorWorkflow` як ViewModel.
Схема повинна показати:
1. Які дані ViewModel отримує з Model (API-відповідь, типи)
2. Які дані ViewModel виставляє у View (що повертає хук)
3. Які команди View може відправити у ViewModel (функції хука)
4. Де живе side-effect (таймер, API-виклик)

### Шаблон схеми

```
Заповни порожні місця:

┌────────────────────────────────────────────┐
│                    VIEW                    │

│                                            │
│  Читає:    stage=?, progress=?, result=?   │  ← що відображає View?
│  Команди:  runAnalyze(), runTailor(), ?    │  ← що може викликати?
└────────────────────────────────────────────┘
              │ читає            │ викликає
              ▼                  ▼
┌────────────────────────────────────────────┐
│               VIEWMODEL                    │
│  useTailorWorkflow()                       │
│                                            │
│  Стан:     [TODO: перелічи всі state]      │
```

## Підсумок

| Патерн | Ключова ідея | Де в React |
|--------|-------------|-----------|
| MVC | Controller між View і Model | Хуки як Controller |
| MVP | Presenter тестується без UI | Хуки + чиста логіка |
| MVVM | ViewModel = реактивне дзеркало | Zustand / useState |

Всі три патерни вирішують одну проблему: **UI не повинен знати де живуть дані і як вони змінюються**.
