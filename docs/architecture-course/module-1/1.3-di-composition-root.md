# 1.3 — DI, IoC та Composition Root

Це фінальний блок Модуля 1. Тут теорія SOLID (особливо DIP) стає практичним патерном.

---

## Три пов'язаних концепти

Їх часто плутають — розберемо по черзі.

```
DIP  ← принцип (WHAT): «залежи від абстракцій, не від реалізацій»
 │
 └─► DI   ← патерн (HOW): «хтось ззовні передає залежність»
              │
              └─► IoC ← механізм (WHO): «фреймворк / контейнер керує lifecycle»
```

---

## Dependency Inversion Principle (DIP) — повторення

З теми 1.1: компонент не повинен **знати** про конкретну реалізацію залежності.

```tsx
// ❌ Порушення DIP — компонент ЗНАЄ що є fetch до /api
import { api } from "../api";
const data = await api.getApplicationData(id);

// ✅ DIP — компонент ЗНАЄ тільки інтерфейс
interface DataLoader {
  getApplicationData(id: string): Promise<ApplicationData>;
}
// Хтось ззовні передасть реалізацію
```

Але хто ж передає реалізацію? І коли? — Це відповідає **Dependency Injection**.

---

## Dependency Injection (DI)

DI — це патерн де **зовнішній код** передає залежність всередину, замість того щоб компонент її сам створював.

### Три способи DI в React

#### 1. DI через props (найпростіший)

```tsx
// Компонент отримує сервіс як звичайний prop
function ApplicationDetail({ service }: { service: ApplicationService }) {
  useEffect(() => {
    service.getApplicationData(id).then(setData);
  }, [id]);
}

// Той хто рендерить — вирішує яку реалізацію передати
<ApplicationDetail service={apiApplicationService} />
// або в тестах:
<ApplicationDetail service={mockApplicationService} />
```

**Плюси**: явно, легко читати, TypeScript гарантує тип
**Мінуси**: prop drilling — якщо сервіс потрібен глибоко, доведеться передавати через всі шари

#### 2. DI через React Context (рекомендований для глобальних сервісів)

```tsx
// Крок 1: контекст визначає інтерфейс (абстракцію)
const ApplicationServiceContext = createContext<ApplicationService | null>(null);

// Крок 2: хук для зручного доступу
function useApplicationService(): ApplicationService {
  const service = useContext(ApplicationServiceContext);
  if (!service) throw new Error("ApplicationServiceContext not provided");
  return service;
}

// Крок 3: компонент не знає звідки сервіс
function ApplicationDetail() {
  const service = useApplicationService();  // ← без props!
  useEffect(() => {
    service.getApplicationData(id).then(setData);
  }, [id]);
}

// Крок 4: десь у дереві — провайдер
<ApplicationServiceContext.Provider value={apiApplicationService}>
  <App />
</ApplicationServiceContext.Provider>
```

**Плюси**: без prop drilling, замінити реалізацію — одне місце
**Мінуси**: неявна залежність (треба знати що контекст існує)

#### 3. DI через кастомний хук (функціональна фабрика)

```tsx
// «Poor man's DI» — хук що інкапсулює залежність
function useApplicationService(): ApplicationService {
  // Тут можна перемикатись між реалізаціями залежно від ENV
  if (import.meta.env.DEV && import.meta.env.VITE_USE_MOCKS) {
    return mockApplicationService;
  }
  return apiApplicationService;
}
```

## Inversion of Control (IoC)

IoC — це ширша ідея: **ти не створюєш залежності — ти оголошуєш їх**.
Хто «контролює» lifecycle об'єктів — не твій код, а **контейнер/фреймворк**.

```
Без IoC:       Компонент сам створює api  → жорстка залежність
З IoC (React): Context.Provider надає api → компонент тільки споживає
```

У React роль IoC-контейнера виконує **React Context** або будь-яка DI-бібліотека.

### Порівняння: з і без IoC у cv-tailor

```tsx
// Без IoC — компонент сам «новить» (або імпортує) залежність
import { api } from "../api";  // ← компонент контролює lifecycle

// З IoC — контейнер управляє lifecycle
const service = useApplicationService();  // ← контейнер контролює lifecycle
// Компонент навіть не знає чи це singleton, чи новий екземпляр
```

## Composition Root — де все збирається

**Composition Root** — це єдине місце в програмі де **всі залежності з'єднуються**.
Всі `new Service()`, всі `createApiClient()`, всі `<Context.Provider value={...}>` — живуть тут, і тільки тут.

### Правило

> Composition Root знаходиться якнайближче до точки входу програми.

У React-додатку — це `main.tsx` або кореневий `App.tsx`.

### Поточний стан cv-tailor: `main.tsx`

```tsx
// client/src/main.tsx — зараз (10 рядків, без Composition Root)
import React from "react";
import ReactDOM from "react-dom/client";
import App from "./App";
import "./styles.css";

ReactDOM.createRoot(document.getElementById("root")!).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>,
);
```

Тут нічого не «збирається» — всі залежності вбудовані прямо в компоненти через `import`.

### Ціль: `main.tsx` як Composition Root

```tsx
// client/src/main.tsx — після рефакторингу
import React from "react";
import ReactDOM from "react-dom/client";
import App from "./App";
import { ApplicationServiceContext } from "./services/ApplicationServiceContext";
import { apiApplicationService } from "./services/ApiApplicationService";
import "./styles.css";

// Composition Root: тут вирішуємо ЯКУ реалізацію використовувати
const service = apiApplicationService;
// або: const service = mockApplicationService; — для розробки без бекенду

ReactDOM.createRoot(document.getElementById("root")!).render(
  <React.StrictMode>
    <ApplicationServiceContext.Provider value={service}>
      <App />
    </ApplicationServiceContext.Provider>
  </React.StrictMode>,
);
```

Тепер щоб переключитись на mock — змінюємо **один рядок** в одному місці.

## Повний приклад: DI для cv-tailor від початку до кінця

### Крок 1: Інтерфейс (абстракція)

```ts
// client/src/services/ApplicationService.ts
import type { Application, ApplicationData, AtsCheckResult, TailoredCv } from "../types";

export interface ApplicationService {
  getApplicationData(id: string): Promise<ApplicationData>;
  patchApplication(id: string, patch: Partial<Application>): Promise<void>;
  runAtsCheck(id: string): Promise<AtsCheckResult>;
  draftFollowupEmail(id: string): Promise<{ email: string }>;
  downloadDocx(cv: TailoredCv): Promise<void>;
  downloadPdf(cv: TailoredCv): Promise<void>;
}
```

### Крок 2: Реалізація (конкретна)

```ts
// client/src/services/ApiApplicationService.ts
import { api } from "../api";
import type { ApplicationService } from "./ApplicationService";

export const apiApplicationService: ApplicationService = {
  getApplicationData: (id) => api.getApplicationData(id),
  patchApplication: (id, patch) => api.patchApplication(id, patch),
  runAtsCheck: (id) => api.runAtsCheck(id),
  draftFollowupEmail: (id) => api.draftFollowupEmail(id),
  downloadDocx: (cv) => api.downloadDocx(cv),
  downloadPdf: (cv) => api.downloadPdf(cv),
};
```

### Крок 3: Контекст (DI-контейнер)

```tsx
// client/src/services/ApplicationServiceContext.tsx
import { createContext, useContext } from "react";
import type { ApplicationService } from "./ApplicationService";

export const ApplicationServiceContext = createContext<ApplicationService | null>(null);

export function useApplicationService(): ApplicationService {
  const service = useContext(ApplicationServiceContext);
  if (!service) {
    throw new Error(
      "useApplicationService must be used within ApplicationServiceContext.Provider"
    );
  }
  return service;
}
```

### Крок 4: Composition Root

```tsx
// client/src/main.tsx
import { ApplicationServiceContext } from "./services/ApplicationServiceContext";
import { apiApplicationService } from "./services/ApiApplicationService";

ReactDOM.createRoot(document.getElementById("root")!).render(
  <React.StrictMode>
    <ApplicationServiceContext.Provider value={apiApplicationService}>
      <App />
    </ApplicationServiceContext.Provider>
  </React.StrictMode>,
);
```

### Крок 5: Споживання в компоненті

```tsx
// client/src/components/ApplicationDetailPanel.tsx
import { useApplicationService } from "../services/ApplicationServiceContext";

// ❌ Більше не потрібно:
// import { api } from "../api";

export function ApplicationDetailPanel({ application }: Props) {
  const service = useApplicationService();  // ← IoC: отримуємо з контейнера

  useEffect(() => {
    service.getApplicationData(application.id).then(setData);
    //      ↑ не знає чи це fetch, чи mock
  }, [application?.id]);
}
```

### Що можна зробити тепер

```tsx
// В тестах або storybook — простий mock без патчинга модулів
const mockService: ApplicationService = {
  getApplicationData: async (id) => ({ /* тестові дані */ }),
  patchApplication: async () => {},
  runAtsCheck: async () => ({ ats_score: 85, /* ... */ }),
  // ...
};

<ApplicationServiceContext.Provider value={mockService}>
  <ApplicationDetailPanel application={testApp} />
</ApplicationServiceContext.Provider>
```

## Low Coupling / High Cohesion — зв'язок з DI

**Low Coupling** (слабке зчеплення): компоненти залежать від мінімальної кількості інших компонентів.

```
❌ Висока зв'язність:
ApplicationDetailPanel → api.ts → fetch → конкретний URL

✅ Низька зв'язність:
ApplicationDetailPanel → ApplicationService (інтерфейс)
                              ↑
                    ApiApplicationService → api.ts → fetch
```

**High Cohesion** (висока зв'язність всередині): все що стосується однієї відповідальності — живе разом.

```
❌ Низька cohesion:
services/
  ApiApplicationService.ts — містить методи і для Application і для MasterCV і для Pipeline

✅ Висока cohesion:
services/
  ApplicationService.ts    — тільки методи для одного Application
  MasterCvService.ts       — тільки методи для MasterCV
  PipelineService.ts       — тільки методи для Pipeline
```

DI + Context дозволяє досягти low coupling: компоненти знають тільки про потрібний їм інтерфейс.
Правильне розбиття інтерфейсів забезпечує high cohesion: кожен сервіс — одна зона відповідальності.

## Підсумок Модуля 1

Ти пройшов повний ланцюжок від принципів до реалізації:

```
SOLID принципи
    ↓ DIP каже: залежи від абстракцій
Dependency Injection
    ↓ DI кажи: хтось ззовні передає залежність
Dependency Inversion через props → через Context
    ↓ IoC каже: контейнер керує lifecycle
React Context як IoC-контейнер
    ↓ Composition Root каже: одне місце для всіх рішень
main.tsx як Composition Root
```

### Що тепер вміє твій код

| До | Після |
|----|-------|
| `ApplicationDetailPanel` жорстко пов'язаний з `fetch` | Залежить від інтерфейсу `ApplicationService` |
| Замінити джерело даних = обшукати 5 компонентів | Замінити одну реалізацію в `main.tsx` |
| Тести вимагають мок fetch | Тести передають `mockApplicationService` |
| Композиція залежностей скрізь | Вся композиція в одному Composition Root |
