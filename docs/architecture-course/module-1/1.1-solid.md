# 1.1 — SOLID у React

SOLID — це 5 принципів, які допомагають писати код, що не перетворюється на кашу при зростанні проєкту.
У React вони не про класи (як у backend OOP), а про компоненти, хуки та інтерфейси.

Для кожного принципу: теорія → живий приклад з cv-tailor → чому болить → як виправити → ДЗ.

---

## S — Single Responsibility Principle

> Кожна одиниця коду має **одну причину для зміни**.

«Причина для зміни» — це конкретний stakeholder або домен, що може попросити про правку.
Якщо компонент одночасно відповідає за форму, за API, за анімацію і за бізнес-логіку — він змінюється з 4 різних причин. Це SRP-порушення.

### ❌ Поганий приклад — `App.tsx` (рядки 91–180)


### Чому це проблема

`App.tsx` має 5 різних «причин змінитися»:
1. Дизайнер хоче інший таб — треба лізти сюди
2. API змінив формат відповіді — треба лізти сюди
3. Логіка tailoring ускладнюється — треба лізти сюди
4. Pipeline потребує нові фільтри — треба лізти сюди
5. Бізнес хоче новий профіль CV — треба лізти сюди

Результат: будь-яка правка ризикує зламати щось непов'язане. Файл стає «страшним місцем».

### ✅ Хороший приклад — розподіл за доменами

### Ключові правила SRP

- [ ] Кожен хук відповідає за **один домен** (workflow / data / UI-state)
- [ ] Компонент **або** рендерить дані, **або** управляє логікою — не обидва
- [ ] Якщо в компоненті > 3 `useEffect` — він, ймовірно, порушує SRP
- [ ] Ознака порушення: ти не можеш назвати компонент одним іменником без «and»

## O — Open/Closed Principle

> Код має бути **відкритий для розширення**, але **закритий для модифікації**.

Тобто: коли потрібна нова функціональність — додаємо новий код, не ламаємо старий.
У React це реалізується через **композицію** замість if-прапорів.

### ❌ Поганий приклад — `ApplicationDetailPanel.tsx` (рядки 159, 291–353)

```tsx
// Стан вибору секції — enum з усіма можливими значеннями
const [activeSection, setActiveSection] = useState<
  "overview" | "cv" | "prep" | "followup" | "interview" | "ats"
>("overview");

// --- Таби (рядки 293–319) ---
// Додати нову секцію = змінити цей компонент
{(["overview", "cv", "prep"] as const).map((s) => (
  <button key={s} className={`detail-tab ${activeSection === s ? "on" : ""}`}
    onClick={() => setActiveSection(s)}>
    {s === "overview" ? "Overview" : s === "cv" ? "Tailored CV" : "Interview Prep"}
  </button>
))}
<button onClick={() => setActiveSection("followup")}>Follow-up</button>
<button onClick={() => setActiveSection("interview")}>Mock Interview</button>
<button onClick={() => setActiveSection("ats")}>ATS Check</button>

// --- Тіло (рядки 332–420+) ---
// Кожна секція — окремий if-блок
{data && activeSection === "overview" && ( <OverviewContent /> )}
{data && activeSection === "cv" && ( <CvPreview cv={data.tailored} /> )}

```

### Чому це проблема

Кожного разу коли продукт хоче **нову вкладку** (наприклад «Salary Research») — треба:
1. Додати значення в тип `activeSection`
2. Додати кнопку в JSX-таби
3. Додати if-блок у тіло
4. Можливо — оновити стан, дефолтне значення

Це 4 місця в одному файлі. Файл **закритий для розширення** — будь-яке розширення вимагає його модифікації.

### ✅ Хороший приклад — конфігурація секцій


### Ключові правила OCP

- [ ] Якщо додавання функції вимагає зміни вже існуючого if/switch — порушення OCP
- [ ] Використовуй **конфіг-масиви** замість switch для варіативної логіки
- [ ] **Slot-pattern** (children або render props) дозволяє розширювати компонент без зміни
- [ ] Ознака порушення: «щоразу коли щось додаю — лізу в той самий файл»

## L — Liskov Substitution Principle

> Якщо B є підтипом A — код, що працює з A, повинен працювати і з B **без жодних змін**.

У React: якщо ти пишеш `<MyButton>` як замінник нативного `<button>` — він повинен приймати **всі** ті самі props: `type`, `disabled`, `aria-*`, `data-*`, `onClick`, тощо.

### ❌ Поганий приклад — wrapper що блокує native props

```tsx
// Уяви що в проєкті є такий компонент (загальна ситуація):
interface ButtonProps {
  label: string;
  onClick: () => void;
  variant?: "primary" | "ghost";
}

function AppButton({ label, onClick, variant = "primary" }: ButtonProps) {
  return (
    <button className={`btn btn-${variant}`} onClick={onClick}>
      {label}
    </button>
  );
}

// Використання:
<AppButton label="Download" onClick={handleDownload} />

// Але якщо треба додати disabled, type="submit", aria-label, form="..." 
// — неможливо! Інтерфейс не пропускає ці props.
<AppButton label="Submit" onClick={submit} disabled={loading} /> // ❌ TypeScript error
```

### Чому це проблема

Компонент `AppButton` нібито замінює `<button>`, але не є повноцінним його замінником.
Кожного разу коли треба нова HTML-властивість — доводиться лізти у визначення `ButtonProps` і вручну додавати.
Компонент порушує «контракт» нативного елемента.

### ✅ Хороший приклад — `ComponentProps` spread

```tsx
import type { ComponentProps } from "react";

// Розширюємо нативний button — не обмежуємо його
interface ButtonProps extends ComponentProps<"button"> {
  variant?: "primary" | "ghost";
}

function AppButton({ variant = "primary", className = "", children, ...rest }: ButtonProps) {
  return (
    <button
      className={`btn btn-${variant} ${className}`.trim()}
      {...rest}  // ← всі нативні props прокидаються автоматично
    >
      {children}
    </button>
  );
}

// Тепер все працює без зміни компонента:
<AppButton variant="primary" disabled={loading} aria-label="Download CV" onClick={handleDownload}>
  Download
</AppButton>
<AppButton type="submit" form="cover-letter-form">
  Generate
</AppButton>
```

### Де це застосовується в cv-tailor

У `ApplicationDetailPanel.tsx` рядок 276:
```tsx
<button className="icon-btn" onClick={onClose} aria-label="Close">×</button>
```
Якщо цю кнопку вирішать винести в окремий `IconButton` компонент — вона має підтримувати `aria-label`, `title`, `disabled` і все інше через LSP-сумісний інтерфейс.

### Ключові правила LSP

- [ ] Wrapper-компоненти над HTML-елементами — завжди `extends ComponentProps<"element">`
- [ ] Завжди прокидуй `...rest` у нативний елемент
- [ ] Не переоголошуй props що вже є в нативному елементі (особливо `onClick`, `type`, `disabled`)
- [ ] Питання для перевірки: «Чи можу я передати `aria-label`? `data-testid`? `form`?»

## I — Interface Segregation Principle

> Клієнт не повинен залежати від методів, які він не використовує.
> Краще багато вузьких інтерфейсів, ніж один широкий.

У React: компонент не повинен отримувати props, які він ніколи не використовує. Якщо `Pipeline` отримує `onDelete` але кожен окремий рядок таблиці не потребує всіх трьох колбеків — є запах ISP-порушення.

### ❌ Поганий приклад — `Pipeline.tsx` (рядки 7–14)

```tsx
// client/src/components/Pipeline.tsx

interface Props {
  applications: Application[];
  onPatch: (id: string, patch: Partial<Application>) => void;   // ← мутація
  onDelete: (id: string) => void;                               // ← видалення
  onOpen: (id: string) => void;                                 // ← відкрити деталі
  loadError: string | null;                                     // ← стан завантаження
  loading: boolean;                                             // ← стан завантаження
}
```

Цей інтерфейс — це «все підряд». `PipelineStats` в середині Pipeline потребує тільки `applications`. Рядок таблиці потребує `onPatch`, `onDelete`, `onOpen` — але не `loadError`. Компонент завантаження потребує `loading` і `loadError` — і нічого іншого.

Кожен споживач залежить від **усього інтерфейсу**, навіть від того, що не використовує.

### ✅ Хороший приклад — розподіл за призначенням

```tsx
// Стан завантаження — окремий інтерфейс
interface PipelineLoadState {
  loading: boolean;
  loadError: string | null;
}

// Дії над рядком — окремий інтерфейс
interface PipelineRowActions {
  onPatch: (id: string, patch: Partial<Application>) => void;
  onDelete: (id: string) => void;
  onOpen: (id: string) => void;
}

// Дані — окремий інтерфейс
interface PipelineData {
  applications: Application[];
}

// Pipeline отримує всі три — але може передати вниз тільки потрібне
interface PipelineProps extends PipelineData, PipelineRowActions, PipelineLoadState {}

// PipelineRow — отримує тільки дані і дії, не знає про loading
interface PipelineRowProps extends PipelineRowActions {
  application: Application;
}

// PipelineLoadingState — отримує тільки стан завантаження
interface PipelineLoadingProps extends PipelineLoadState {}
```

Переваги:
- Кожен компонент залежить тільки від того що йому потрібно
- Тести легше писати — не треба мокати весь об'єкт
- Якщо `onDelete` зміниться — перекомпілюються тільки компоненти що його справді використовують

### Де ще порушується ISP у cv-tailor

`statusClass()` функція **продубльована** в `Pipeline.tsx` (рядок 16) і `ApplicationDetailPanel.tsx` (рядок 18) — кожен файл включає логіку, яку міг би взяти з одного місця. Це інша форма ISP-порушення: відсутність спільного інтерфейсу.

### Ключові правила ISP

- [ ] Якщо компонент отримує prop і ніколи його не використовує — порушення ISP
- [ ] Розбивай великі Props-інтерфейси на окремі за призначенням
- [ ] Дублювання однакових функцій у різних файлах — сигнал що треба спільний модуль
- [ ] Питання для перевірки: «Чи є хоч один prop що цей компонент не використовує?»

## ДЗ 4 — Розбити `Props` у `Pipeline.tsx`

### Завдання

У `Pipeline.tsx` рядки 7–14 — один великий `Props` інтерфейс.
Твоє завдання:

## D — Dependency Inversion Principle

> Модулі **верхнього рівня** не повинні залежати від модулів **нижнього рівня**.
> Обидва повинні залежати від **абстракцій** (інтерфейсів).

Простіше: компоненти не повинні знати **де** беруться дані — через `fetch`, `localStorage`, Firebase чи mock. Вони мають знати тільки **що** їм потрібно (інтерфейс).

### ❌ Поганий приклад — `ApplicationDetailPanel.tsx` (рядки 2, 189–192)

```tsx
// Рядок 2 — жорстка залежність від конкретної реалізації
import { api } from "../api";

// Рядки 189–192 — використання прямо в компоненті
useEffect(() => {
  api
    .getApplicationData(application.id)
    .then(setData)
    .catch((e) => setLoadError(e.message));
}, [application?.id]);
```

### Чому це проблема

`ApplicationDetailPanel` **знає** що є `fetch`-based API на `/api/*`.
Якщо:
- В тестах треба mock-дані — неможливо без патчинга модуля
- Переходиш на GraphQL — треба шукати всі компоненти де `api.*` викликається
- Хочеш offline-режим — немає місця де підмінити джерело даних
- Хочеш storybook з тестовими даними — знову патчинг

### ✅ Хороший приклад — залежність через абстракцію

```tsx
// Крок 1: визначаємо інтерфейс (абстракцію)
// client/src/services/ApplicationService.ts

export interface ApplicationService {
  getApplicationData(id: string): Promise<ApplicationData>;
  patchApplication(id: string, patch: Partial<Application>): Promise<void>;
  runAtsCheck(id: string): Promise<AtsCheckResult>;
  draftFollowupEmail(id: string): Promise<{ email: string }>;
  downloadDocx(cv: TailoredCv): Promise<void>;
  downloadPdf(cv: TailoredCv): Promise<void>;
}

// Крок 2: компонент залежить від абстракції, не від реалізації
interface Props {
  application: Application | null;
  onClose: () => void;
  onPatch: (id: string, patch: Partial<Application>) => void;
  service: ApplicationService;  // ← інжектована залежність
}

export function ApplicationDetailPanel({ application, service, onClose, onPatch }: Props) {
  useEffect(() => {
    service.getApplicationData(application.id).then(setData);
    //      ↑ не знає чи це fetch, чи mock, чи IndexedDB
  }, [application?.id]);
}

// Крок 3: реальна реалізація (в api.ts або окремому файлі)
// client/src/services/ApiApplicationService.ts
import { api } from "../api";

export const apiApplicationService: ApplicationService = {
  getApplicationData: api.getApplicationData,
  patchApplication: api.patchApplication,
  runAtsCheck: api.runAtsCheck,
  draftFollowupEmail: api.draftFollowupEmail,
  downloadDocx: api.downloadDocx,
  downloadPdf: api.downloadPdf,
};

// Крок 4: App.tsx (або Composition Root) знає яку реалізацію передати
<ApplicationDetailPanel
  application={selectedApp}
  service={apiApplicationService}
  onClose={closeDetail}
  onPatch={pipeline.patch}
/>
```

### Ключові правила DIP

- [ ] Компоненти не роблять `import { api }` — вони отримують сервіс ззовні
- [ ] Інтерфейс описує **що** (методи і типи), реалізація описує **як** (fetch, IndexedDB)
- [ ] Один компонент — один service-інтерфейс (не «весь API»)
- [ ] Питання для перевірки: «Якщо замінити fetch на mock — скільки файлів треба змінити?»
